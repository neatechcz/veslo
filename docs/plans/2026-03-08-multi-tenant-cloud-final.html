Veslo Multi-Tenant Cloud Final Spec

Final Draft MVP-first design for true multi-tenancy with organization ownership, platform admin role separation, and strict org isolation.

1. Core Decisions

Area Final Decision MVP Status
Tenant model All business entities are organization-scoped via org_id; requests require active org context. In MVP
Roles platform_admin (global), org_owner, org_member. In MVP
Workers naming UI label changes from Workers to Projects; backend can keep worker entity initially. In MVP
Project creation Any org user can create projects in MVP. In MVP
Admin authority Admin has stronger rights than owners, but used for exceptional interventions (security/compliance/recovery), not day-to-day operations. In MVP
Private member-only projects Do not block future support for owner-inaccessible private projects; not implemented in first version. Post-MVP
AI project audits Future AI guardrails may warn/stop projects based on company policy; not part of first version. Post-MVP

2. Domain Login and Organization Creation

Login Condition User Options Owner Policy Influence Outcome
No organization exists for domain Create new organization N/A User becomes first owner.
Organization exists for domain Create new org or join existing org Owner can lock domain behavior Default open choice unless locked.
Domain is locked by owner for join-only Join existing only Owner-defined domain lock New org creation disabled for matching domain.
Join requires approval Submit join request Owner approval workflow User becomes member after approval.
Public domains (gmail/outlook/etc.) Create personal org by default or join via invite/request Public-domain list is advisory only No dependency on keeping a perfect public-domain list.

3. Access and Control Matrix

Capability Org Member Org Owner Platform Admin
Create project Yes Yes Yes
Manage project access/policy Own or assigned projects only Any project in org Global authority
Stop project (normal operation) Own/assigned projects Any org project Yes, but intended for exceptional interventions
Suspend organization execution No No Yes (security/compliance/billing incidents)
Manage org members and roles No Yes Yes
View/edit platform settings No No Yes only
View/edit org settings Read/use only (configurable) Yes Yes

Admin interventions must always create audited records with reason, actor, scope, and timestamp.

4. Settings Boundaries

Settings Scope Examples Visibility Edit Rights
Platform settings Global model provider credentials, global model catalog, platform connectors. Platform admins only Platform admins only
Organization settings Join policy, domain lock, approval requirements, org model preference profile. Owners (optional member read-only) Owners
Project settings Project defaults, runtime knobs, project-level assets. Project collaborators Creator/assigned collaborators + owner
User settings Personal preferences and private artifacts. User only User only

5. Billing Model (Organization-Owned)

Billing Rule Decision MVP Status
Subscription cardinality Exactly one subscription per organization. In MVP
Billing responsibility Organization owners manage billing. In MVP
Self-serve mode Owner pays through Polar/Stripe checkout. In MVP
Admin invoiced mode Owner requests; admin approves; invoice handled outside in-app card flow. In MVP
Trial control Admin can set or extend trial duration to any date/time. In MVP

6. Artifact Sharing Semantics

Concept Final Meaning MVP Status
Default artifact scope Everything starts private (user-private or org-private). In MVP
Sharing model Sharing is an explicit overlay action in UI, not default state. In MVP
Member contribution Member creates skill/MCP/automation and submits to owner for org sharing approval. Post-MVP (can stub workflow in MVP)
Org-to-global promotion Owner proposes org artifact to global catalog; admin approves before global rollout. Post-MVP

Function Purpose Why Required for V1 Multi-Tenancy
resolveUserOrganizations(userId)List org memberships for current user.Needed for org switch and auth scope.

approveOrganizationJoin(orgId, ownerId, requestId)Owner accepts join request.Owner-controlled membership.
inviteOrganizationMember(orgId, ownerId, email, role)Owner invites member/owner.Controlled org growth.

9. Strict MVP Cut (Removed for Now)

Deferred Capability Reason Deferred
Global artifact catalog publication workflowNot required to prove tenant isolation and org/admin separation.
Member-to-owner artifact contribution workflow UICan ship after core tenant/billing foundation.
Org-to-global promotion approval pipelineDepends on global artifact catalog capabilities.
Automated DNS domain verificationManual/domain policy baseline is enough for MVP.
AI project policy audits and auto-stopFuture governance layer, not core multi-tenancy.
Owner-inaccessible private member projectsFuture privacy model; keep data model extensible but skip V1.
