Veslo Multi-Tenant Cloud: Planned vs Implemented

Single Comparison Document Full planned multi-tenant model compared against the actual implementation built so far.

This document is intentionally not split into MVP vs final specs. It compares the full intended model against the current implementation state.

Implementation state reflects the work completed on branch feature/multi-tenant-mvp at commit 72b0df08.

Status legend: Implemented Partial Not implemented

1. Operating Model Comparison

Area Planned Model Actual Implementation Status Difference / Next Step
Organization creation Users can create organizations; when no matching organization exists for their domain, they create a new org and become the first owner. Users without memberships are given a personal/default org via bootstrap logic. There is no general create-organization flow yet. Partial Add explicit organization creation beyond personal bootstrap.
Domain-based onboarding After sign-in, domain match drives `create new org` vs `join existing org` based on owner-controlled domain rules. No org_domain model, no onboarding decision endpoint, no create-vs-join flow, and no join requests. Not implemented Add org_domain, onboarding decision API, and join request flow.
Public-domain behavior Public domains should default to personal org creation, while still allowing invite/request-based joining. The public-domain list is advisory only. Users with no memberships fall back to a personal org, but there is no explicit public-domain classification or invite/request UX. Partial Add advisory public-domain handling as part of domain onboarding.
Role model Three roles: platform_admin, org_owner, org_member, with admin stronger than owner. platform_role exists for admin grants. Org membership owner/member roles exist. Admin can bypass owner checks on org routes and delete any worker. Partial Role model exists in foundation, but not all planned admin-controlled surfaces exist yet.
Active org context Every multi-tenant request uses explicit active_org_id and users can switch among organizations they belong to. Cloud control loads orgs, stores selected org in browser local storage, and sends x-veslo-org-id. Single-org users get automatic fallback. Partial Current active org is client-managed, not server session-managed.
Tenant isolation All tenant data is keyed by org_id and filtered by org context. Membership and worker routes are org-scoped. Future entities are not present yet. Partial Isolation works on current org-aware surfaces; extend the same rule to future entities.
Project creation and naming UI should use `Projects`; members can create them; backend may temporarily keep worker. Members can create org-scoped workers. UI still uses `Workers` in visible labels. Partial Rename UI from Workers to Projects.
Private member-only projects Keep the door open for member-private projects that owners cannot access, but do not implement this in the first version. No private-project model exists. Not implemented Future privacy model only; not part of the current implementation.
Admin superuser intervention Admin has stronger-than-owner authority for exceptional cases only: legal, security, billing, compliance, or recovery. Actions must be audited. Admin can bypass owner-only membership checks and delete any org worker. There are no project/org suspend controls yet. Partial Add explicit stop/suspend/recovery controls with audit requirements.

2. Governance, Settings, Billing, and Sharing Comparison

Area Planned Model Actual Implementation Status Difference / Next Step
Platform settings boundary Platform settings are admin-only and hidden from owners and members. No platform settings surface exists yet, so owners/members do not see one, but the feature itself is not implemented. Partial Add explicit platform settings table/API/UI for admins only.
Organization settings Owners manage org join rules, domain policy, and future org-specific model/provider preferences. No org_settings model or settings API exists. Not implemented Add org_settings and owner settings surfaces.
Org model preferences Each organization can pick allowed provider/model defaults such as OpenAI or Gemini within platform policy. No organization-level model preference system exists. Not implemented Implement this through org_settings.
Billing ownership Exactly one subscription per org, managed by org owners. Billing is still user-scoped under worker billing endpoints. Not implemented Move billing ownership to org_subscription.
Self-serve billing Owners start Stripe/Polar checkout for the organization subscription. Checkout exists only in the user-scoped worker billing flow. Partial Re-scope checkout to organization billing.
Admin-invoiced billing Owners can request invoiced billing; admin approves and the org runs without in-app card entry. No invoiced billing workflow exists. Not implemented Add org_billing_request and admin approval flow.
Admin trial control Admin can set or extend trial to any date/time for any org. No organization trial control exists. Not implemented Add org trial fields and admin controls.
Artifact default scopes Artifacts default to user-private or org-private; sharing is an explicit overlay action. No artifact-sharing model exists yet. Not implemented Add artifact data model and default private scope semantics.
Org-level sharing Owners can publish MCPs, skills, and automations to their organization. No org-sharing workflow exists. Not implemented Add publish and approval workflow for org scope.
Member contributions Members can create private skills/MCPs/automations and submit them to owners for org sharing. No member contribution workflow exists. Not implemented Add submission, approval, and revision states.
Org-to-global promotion Owners can submit org artifacts for global sharing and admins approve or reject. No promotion workflow exists. Not implemented Add global catalog approval path.
AI audits / auto-stop Future AI governance can warn or stop projects based on organization policy. No AI audit or auto-stop system exists. Not implemented Future governance layer only.

3. Data Model Comparison

Table / Entity Planned Model Actual Implementation Status Difference / Next Step
org Organization root with tenant policy and billing-related configuration. Exists and is used as the tenant root. Implemented Extend with planned domain and billing policy fields as needed.
org_membership Owner/member membership model with multi-owner support. Exists and is used. Owner/member role enforcement is active. Final-owner protection exists. Implemented Invitation and join-request layers still need to be added around it.
platform_role Global platform role grants, especially platform_admin. Implemented and used for admin capability checks. Implemented No bootstrap/admin-management UI yet.
org_domain Domain-to-organization mapping with join/create rules and later verification. Does not exist. Not implemented Required for domain onboarding.
org_join_request Owner-managed request queue for domain joins. Does not exist. Not implemented Required for approval-based domain joining.
org_invite Invitation token/email flow for invite-only onboarding. No invitation table or invite token flow exists. Not implemented Current member add requires the target user to already exist.
org_settings Organization-level settings including join policy and model preferences. Does not exist. Not implemented Needed for owner-managed org configuration.
org_subscription Single subscription record per organization. Does not exist. Not implemented Needed to move billing from user scope to org scope.
org_billing_request Admin-approved invoiced billing workflow. Does not exist. Not implemented Needed for invoice-based billing mode.
platform_settings Admin-only global provider credentials and infrastructure settings. Does not exist. Not implemented Needed for a real admin configuration surface.
worker Temporary backend runtime entity behind the user-facing Project concept. Exists and is org-scoped. Used for list/create/get/token/delete. Implemented UI naming still needs to move to Projects.
audit_event Immutable audit trail for membership, billing, admin actions, and publishing workflows. Exists and is used for membership and worker governance events. Partial Coverage needs to expand as more admin, billing, and sharing actions are added.
artifact, artifact_release, artifact_distribution Artifact sharing, approval, and global rollout model. Do not exist. Not implemented Needed for MCP/skill/automation sharing.

startOrganizationCheckout(orgId, ownerId) Start self-serve organization checkout. Current checkout exists only for user-scoped worker billing. Partial Re-scope checkout to organization ownership.
requestInvoicedBilling(orgId, ownerId) Request invoice-based billing. Not implemented. Not implemented Add invoice request workflow.
approveInvoicedBilling(orgId, adminId) Admin approves invoiced billing. Not implemented. Not implemented Add admin invoice approval flow.
