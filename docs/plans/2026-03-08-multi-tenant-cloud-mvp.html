Veslo Multi-Tenant Cloud MVP

Invite-First MVP This is the reduced first version we can implement without taking on domain automation and billing rewrites.

1. MVP Scope

Area MVP Decision
Tenant modelAll org-scoped data stays keyed by org_id, and requests must carry explicit org context.
Rolesplatform_admin, org_owner, org_member.
Signup behaviorKeep existing personal-org auto-creation on signup.
Multi-org accessOwner-managed membership only. No domain matching in MVP.
ProjectsMembers can create projects. Cloud control exposes an active-organization selector for multi-org users.
Platform settingsAdmin-only visibility and edit access.
BillingKeep current billing implementation unchanged for this MVP.

2. Not in MVP

Deferred Capability Reason
Domain create/join routingNot required to ship real tenant separation.
org_domainCan come after invite-first foundation.
org_join_requestApproval queue not needed if owners add members directly.
Org model preference settingsUseful later, not required for V1 multi-tenancy.
Org-owned billing rewriteCurrent billing is user-centric and would slow the first release.
Admin-invoiced billingEnterprise billing path can follow the tenant foundation.
Artifact sharing workflowsSeparate feature family.
AI audit and auto-stopFuture governance layer.
Private member-only projectsKeep extension path open, but do not build now.
Workers -> Projects renameNot required to ship the first working multi-tenant release.

3. Required Technical Changes

Change What It Means
Explicit org contextStop using “first membership wins”; resolve org from request and validate membership.
Admin role modelAdd global admin grants separate from org roles.
Organization listing/switchingUsers with multiple org memberships need a list of orgs and a way to select active org.
Owner-managed membershipOwners add/remove members and promote/demote owners.
Project scopingProject list/create/get/delete/token APIs must always be filtered and authorized by active org.
Audit coverageMembership changes and admin interventions must be logged.

6. Acceptance Criteria

Requirement Expected Result
User belongs to two orgsUser can switch org and see different projects in cloud control.
Cross-org accessUser cannot read or mutate projects outside their memberships.
Owner membership managementOwner can add another existing user into org, change their role, and avoid leaving the org ownerless.
Admin authorityPlatform admin can bypass owner checks for exceptional operations.
Migration safetyExisting single-org users continue to work.
